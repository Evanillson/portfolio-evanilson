# Como construir um Star Schema eficiente para Power BI

A modelagem de dados é o alicerce de qualquer solução de Business Intelligence. Antes de pensar em visuais bonitos ou fórmulas DAX complexas, você precisa de uma base sólida — e essa base é o **Star Schema**.

## Por que Star Schema?

O Star Schema (esquema estrela) é o padrão recomendado pela Microsoft para o Power BI. Mas por quê?

- **Performance**: O motor VertiPaq do Power BI é otimizado para comprimir e consultar dados em formato colunar. Tabelas dimensionais estreitas e tabelas fato numéricas são o cenário ideal.
- **Simplicidade**: Relações claras entre fato e dimensões eliminam ambiguidades nos cálculos DAX.
- **Manutenção**: Adicionar novas métricas ou filtros se torna trivial quando o modelo é bem estruturado.

## Anatomia de um Star Schema

```
          ┌──────────┐
          │ dim_data  │
          └─────┬────┘
                │
┌──────────┐    │    ┌──────────────┐
│dim_cliente├────┼────┤  fato_vendas │
└──────────┘    │    └──────┬───────┘
                │           │
          ┌─────┴────┐ ┌────┴───────┐
          │dim_produto│ │dim_regional│
          └──────────┘ └────────────┘
```

### Tabela Fato
A tabela fato contém as **métricas numéricas** (valor_venda, quantidade, custo) e as **chaves estrangeiras** que conectam às dimensões. Ela deve ser:
- Granular (uma linha por transação)
- Estreita (apenas FKs e medidas)
- Grande em linhas, pequena em colunas

### Tabelas Dimensão
As dimensões contêm os **atributos descritivos** que contextualizam as métricas: nome do cliente, categoria do produto, data. Elas devem ser:
- Desnormalizadas (flattened)
- Enriquecidas com hierarquias
- Conectadas à fato por relações 1:N

## Erros Comuns a Evitar

### 1. Snowflaking excessivo
Normalizar as dimensões (criar sub-dimensões) pode parecer elegante do ponto de vista relacional, mas **mata a performance** no Power BI. Flatten é a palavra de ordem.

### 2. Relações muitos-para-muitos
Se você tem uma relação M:N, crie uma **bridge table** (tabela ponte). Nunca force uma relação M:N diretamente — o DAX se comporta de forma imprevisível.

### 3. Colunas calculadas na tabela fato
Evite adicionar colunas calculadas em tabelas fato. Elas consomem RAM e geralmente podem ser substituídas por **medidas** (que são calculadas on-the-fly).

## Implementação Prática

```sql
-- Exemplo de ETL para criar a dimensão de data
CREATE TABLE dim_data AS
SELECT DISTINCT
    CAST(data_venda AS DATE) AS data_id,
    YEAR(data_venda) AS ano,
    MONTH(data_venda) AS mes,
    DAY(data_venda) AS dia,
    DATENAME(WEEKDAY, data_venda) AS dia_semana,
    CASE WHEN MONTH(data_venda) <= 6 THEN 'S1' ELSE 'S2' END AS semestre
FROM fato_vendas_raw;
```

```dax
// Medida DAX que aproveita o star schema
Vendas YTD =
CALCULATE(
    SUM(fato_vendas[valor_venda]),
    DATESYTD(dim_data[data_id])
)
```

## Conclusão

Investir tempo na modelagem é investir em escalabilidade. Um modelo bem feito no início evita retrabalho, melhora performance e torna o DAX intuitivo. O Star Schema não é apenas uma recomendação — é a fundação para qualquer projeto de BI sério.

> "Dados sem modelo são apenas ruído. O modelo é o que transforma ruído em inteligência."
